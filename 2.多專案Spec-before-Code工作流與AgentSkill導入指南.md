# Spec-before-Code 多專案導入指南 — Prompt 範例集

> 建立日期: 2026-03-10
> 用途: 可套用到任何使用 Claude Code 的多專案結構
> 範例專案: xcomply.ai / TextComply

---

## 專案背景

以 xcomply.ai（TextComply）為例，這是一個合規科技平台，採多資料夾結構：

| 資料夾 | 角色 |
| --- | --- |
| `Iqt.TextComply.Frontend.React` | 前台前端（React） |
| `Iqt.TextComply.Backend` | 前台後端（.NET / Node.js） |
| `Iqt.TextComply.Admin` | 後台管理 |
| `Iqt.TextComply.Schema` | 資料庫結構（SQL / Migration） |

---

## Claude Code 正確目錄結構

```text
{project-root}/

│   └── skills/                            # 按需呼叫的擴充技能

├── docs/                                  # 共用文件（跨專案）
│   ├── README.md
│   ├── specs/                             # 技術規格文件
│   │   ├── _spec-template.md
```

### 各目錄角色對照

| 目錄 | 載入方式 | 用途 |
| --- | --- | --- |
| `CLAUDE.md` | 啟動自動載入 | 專案總覽、強制性規則、資料夾說明 |
| `.claude/commands/` | 使用者輸入 `/指令名` 觸發 | 自訂 Slash 指令（如 `/create-spec`） |
| `.claude/rules/` | 啟動自動載入 | 細部常駐規則（補充 CLAUDE.md） |
| `.claude/skills/` | AI 依情境按需呼叫 | 專業工作流技能模組 |
| `docs/specs/` | 開發流程中引用 | Spec-before-Code 技術規格 |
| `docs/bugs/` | 開發流程中引用 | Bug 知識庫 |
| `plans/` | 開發流程中引用 | 功能開發計劃書 |

## PROMPT 範例

---

### 1. 依據新增 command 更新 README 及 CLAUDE.md

```
我新增了一個 command: `xxx`，
請幫我更新 README.md 和 CLAUDE.md，
把這個 command 的用途與用法補上去，格式跟現有的一致。
```

### 2. 依據新增 command 更新 Agent Skill

```
我新增了一個 command: `xxx`，
請幫我更新 .claude/skills/ 裡對應的 skill 文件，
新增對應的 skill 描述，格式跟現有的一致。
```

### 3. 一鍵導入：為多專案建立 Spec-before-Code 工作流

```
我的專案是多資料夾結構，每個資料夾是獨立的子專案：
- {Frontend 資料夾名} => 前台前端
- {Backend 資料夾名} => 前台後端
- {Admin 資料夾名} => 後台
- {Schema 資料夾名} => 資料庫結構

請幫我建立完整的 Spec-before-Code 工作流：
1. CLAUDE.md — 專案總覽與強制性工作流協議
2. .claude/commands/_spec-convention.md — Spec 撰寫慣例
3. .claude/commands/create-spec.md — /create-spec 指令
4. .claude/rules/spec-before-code.md — 常駐強制規則
5. .claude/skills/multi-project-workflow.md — 多專案協作技能
6. docs/specs/_spec-template.md — 多專案版 Spec 模板（含「受影響子專案」表格）
7. docs/bugs/_bug-template.md — Bug 記錄模板
8. 各目錄的 README.md
9. plans/ 目錄結構

規則：
- 超過 3 行的改動必須先建 Spec，使用者確認後才能寫程式碼
- 跨子專案改動必須列出所有受影響的子專案
- 開發順序建議：Schema → Backend → Frontend → Admin
- 完成後 Spec 從 pending/ 搬到 completed/
```

### 4. 為現有專案產生 CLAUDE.md 指引文件

```
請幫我掃描這個工作目錄的結構，
依據現有的資料夾、檔案命名慣例、技術棧，
產生一份 CLAUDE.md，包含：

1. 專案概述（一句話描述）
2. 各資料夾用途說明（表格）
3. 技術棧與開發環境
4. 開發慣例（命名規則、分支策略、commit 格式）
5. 常用指令（build / test / deploy）
6. 強制性工作流協議（Spec-before-Code）

格式跟現有慣例一致。
```

### 5. 為現有專案產生 Agent Skill

```
請幫我依據這個工作目錄的程式碼脈絡，
在 .claude/skills/ 建立技能文件，用來教 AI 怎麼協作這個專案。

內容需包含：
1. 專案架構總覽（哪些子專案、各自職責）
2. 跨專案關聯性（前後端怎麼串、Schema 怎麼影響其他專案）
3. AI 協作技能清單（每個 skill：名稱、觸發時機、執行步驟、注意事項）
4. 常見任務的 prompt 範例

Skill 至少涵蓋：
- 新增 API endpoint（後端 + 前端串接）
- 新增資料表（Schema → Backend → Frontend 連動）
- 修 Bug（定位 → Spec → 修復 → 驗證）
- 重構（影響範圍分析 → Spec → 執行）
```

### 6. 跨子專案功能開發（全端聯動）

```
我要新增一個功能：{功能名稱}，
這個功能會跨越多個子專案：
- Schema: {需要新增/修改的資料表}
- Backend: {需要新增的 API}
- Frontend: {需要新增的頁面或元件}
- Admin: {需要新增的後台管理頁面}（如果有的話）

請幫我：
1. 先建立 Spec（docs/specs/pending/{feature-name}.spec.md）
2. 列出每個子專案受影響的檔案與邏輯變更點
3. 建議開發順序（Schema → Backend → Frontend → Admin）
4. 等我確認後再開始開發
```

### 7. 分析現有程式碼建立開發慣例

```
請幫我分析 {資料夾名} 的程式碼，找出：
1. 命名慣例（檔案名、變數名、函式名的 pattern）
2. 目錄結構慣例（檔案怎麼分類放置）
3. 常用的 design pattern（Repository、Service Layer 等）
4. 錯誤處理方式
5. API 回應格式

然後在 .claude/rules/ 建立對應的慣例規則文件，
讓 AI 未來寫程式碼時自動遵循這些慣例。
```

### 8. Bug 修復標準化流程

```
我遇到一個 Bug：{Bug 描述}

請依照以下流程處理：
1. 先分析可能的根本原因（不要直接改 code）
2. 建立 Spec（docs/specs/pending/fix-{bug-name}.spec.md）
3. 在 Spec 中列出：問題描述、重現步驟、根因分析、修復方案、預期測試結果
4. 等我確認後再開始修復
5. 修復完成後更新 Spec 的 Bug Log
6. 若 Bug 有通用參考價值，另存到 docs/bugs/
```

### .claude/rules/spec-before-code.md

```markdown
# Spec-before-Code 強制規則

## 觸發條件

任何超過 3 行的程式碼改動，必須：

1. 先在 `docs/specs/pending/` 建立 Spec 文件
2. 使用 `docs/specs/_spec-template.md` 模板
3. 向使用者展示 Spec 內容，等待確認
4. 確認後才開始寫程式碼
5. 完成後將 Spec 移至 `docs/specs/completed/`

## 跨專案改動（額外規則）

若改動涉及多個子專案：
- Spec 必須包含「受影響子專案」表格
- 必須列出建議開發順序（通常：Schema → Backend → Frontend → Admin）
- 每個子專案的受影響檔案必須分開列出

## 豁免

以下不需建 Spec：修正 typo、更新版本號、修改設定值、純格式化調整
```

### .claude/skills/multi-project-workflow.md

```markdown
# Skill: 多專案協作工作流

## 專案架構

| 子專案 | 角色 |
| --- | --- |
| Iqt.TextComply.Schema | 資料庫結構（最上游） |
| Iqt.TextComply.Backend | 前台後端 |
| Iqt.TextComply.Frontend.React | 前台前端 |
| Iqt.TextComply.Admin | 後台管理 |

## 跨專案關聯性

- Schema 變更 → 必須同步 Backend Model → 可能影響 Frontend API 呼叫
- Backend API 變更 → 必須同步 Frontend 串接
- 任何資料表新增/修改 → 檢查 Admin 是否需要對應的管理介面

---

## Skill 1: 新增 API Endpoint（全端聯動）

**觸發**: 使用者要求新增 API 或功能需要新的後端接口
**步驟**:
1. 確認是否需要新資料表（→ 先處理 Schema）
2. 在 Backend 新增 Controller / Service / Repository
3. 在 Frontend 新增 API 呼叫與對應 UI
4. 確認 Admin 是否需要管理頁面
5. 全程遵循 Spec-before-Code

## Skill 2: 新增資料表（Schema 起手）

**觸發**: 使用者要求新增或修改資料表
**步驟**:
1. 在 Schema 新增 migration 檔案
2. 在 Backend 新增 Model / DTO
3. 在 Backend 新增 Repository 與 Service
4. 視需求在 Frontend / Admin 新增介面
**注意**: migration 順序在 Spec 中明確記錄

## Skill 3: 修復 Bug

**觸發**: 使用者回報 Bug
**步驟**:
1. 分析根本原因（先讀 code，不直接改）
2. 判斷影響範圍（是否跨子專案）
3. 建立 Spec，列出修復方案
4. 使用者確認後修復
5. 更新 Bug Log

## Skill 4: 重構

**觸發**: 使用者要求重構模組或改善品質
**步驟**:
1. 分析問題點
2. 評估影響範圍（grep 所有引用點）
3. 建立 Spec
4. 使用者確認後逐步執行
**注意**: 重構不改變外部行為，Spec 需列出不變性約束

## Skill 5: 更新文件與指引

**觸發**: 新增了 command、功能、或改了架構
**步驟**:
1. 更新 CLAUDE.md
2. 更新 .claude/skills/ 對應技能文件
3. 更新 .claude/commands/ 對應指令（如有）
4. 更新 README.md
5. 格式跟現有慣例一致
```

### .claude/commands/_spec-convention.md

```markdown
# 技術規格文件（Spec）撰寫慣例

所有超過 3 行的程式碼改動，在開始寫程式碼之前，必須先建立技術規格文件。

---

## 強制性規則

1. **超過 3 行的改動 = 必須建立 Spec**
2. **Spec 必須經使用者確認後才能開始寫程式碼**
3. **開發過程中遇到 Bug，必須記錄在 Spec 的 Bug Log 區塊**
4. **開發完成後，Spec 從 `pending/` 移動到 `completed/`**
5. **跨子專案改動，必須列出所有受影響的子專案與開發順序**

### 豁免清單

不需建 Spec：修正 typo、更新版本號、修改設定值、純格式化調整

---

## 檔案規範

- **位置**: `docs/specs/pending/[task-name].spec.md`
- **命名**: 英文 kebab-case（如 `add-user-crud.spec.md`）
- **模板**: `docs/specs/_spec-template.md`

---

## 生命週期

1. 建立 Spec → `pending/`（🟡 待確認）
2. 使用者確認（🔵 開發中）
3. 依 Spec 逐步實作
4. 遇問題寫入 Bug Log
5. 完成歸檔 → `completed/`（✅ 已完成）

---

## Bug Log 格式

### Bug #{序號}: {問題簡述}

| 分類 | 內容 |
| --- | --- |
| **[Bug]** | {問題描述} |
| **[Root Cause]** | {根本原因} |
| **[Solution]** | {解決方案} |
| **[Prevention]** | {預防措施} |

通用 Bug 另存 `docs/bugs/`。
```

### .claude/commands/create-spec.md

```markdown
# 建立技術規格文件（Spec）

為即將進行的改動建立技術規格文件。

> **必須遵循** `.claude/commands/_spec-convention.md`

---

## 執行步驟

### Phase 1: 需求收集

1. 詢問改動內容（功能名稱、目的）
2. 確認範圍（受影響的子專案）
3. 確認是否有關聯的計劃書（plans/）

### Phase 2: 分析受影響檔案

1. 列出所有會被修改或新增的檔案（按子專案分類）
2. 說明具體的邏輯變更點
3. 評估風險與副作用
4. 建議開發順序（Schema → Backend → Frontend → Admin）

### Phase 3: 建立 Spec 文件

1. 使用 `docs/specs/_spec-template.md` 模板
2. 在 `docs/specs/pending/` 建立 `[task-name].spec.md`
3. 填入所有區塊

### Phase 4: 等待確認

1. 展示 Spec 摘要
2. 詢問使用者是否確認
3. 確認後才開始開發
```
