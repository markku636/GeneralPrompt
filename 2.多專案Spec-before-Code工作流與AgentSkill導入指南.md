# Spec-before-Code 一鍵導入 Prompt

> 使用方式：到目標專案根目錄開啟 Claude Code，複製「一鍵導入」區塊貼上即可。
> 唯一要改的：把 `{你的專案描述}` 換成實際描述（例如「合規科技平台」「電商系統」）。

---

## 一鍵導入

複製以下內容，貼到 Claude Code：

````text
請幫我建立 Spec-before-Code 工作流。

專案描述：{你的專案描述}

### 你要做的事

1. 掃描這個工作目錄，辨識所有子專案資料夾
   - 排除 docs/、plans/、.claude/、node_modules/、.git/、bin/、obj/ 等非專案目錄
   - 從 package.json、*.csproj、*.sln、requirements.txt、go.mod 等判斷每個子專案的角色與技術棧
   - 推斷子專案之間的依賴關係與建議開發順序

2. 依據掃描結果，產生以下所有檔案：

---

#### 檔案 1: CLAUDE.md

```markdown
# {專案名稱} — Claude Code 指引

## 專案概述

{專案描述}，採用多專案結構。

## 子專案說明

| 資料夾 | 角色 | 技術棧 |
| --- | --- | --- |
| `{掃描結果}` | {掃描結果} | {掃描結果} |

## 共用文件

| 資料夾 | 用途 |
| --- | --- |
| `docs/specs/` | 技術規格文件（Spec-before-Code） |
| `docs/bugs/` | Bug 知識庫 |
| `docs/decisions/` | 技術決策記錄 |
| `plans/` | 功能開發計劃書 |

## 強制性工作流協議（Spec-before-Code）

**所有超過 3 行的程式碼改動**，必須先建立技術規格文件，等使用者確認後才能寫程式碼。

### 流程

建立 Spec → 使用者確認 → 開始開發 → 遇 Bug 記錄 → 完成歸檔

### 多專案規則

- 跨子專案的改動，Spec 必須列出所有受影響的子專案
- 開發順序：{依掃描結果推斷}
- 慣例文件：`.claude/commands/_spec-convention.md`
- 快速建立：`/create-spec`

### 豁免清單

不需建 Spec：修正 typo、更新版本號、修改設定值、純格式化調整
```

#### 檔案 2: .claude/rules/spec-before-code.md

```markdown
# Spec-before-Code 強制規則

## 觸發條件

任何超過 3 行的程式碼改動，必須：

1. 先在 `docs/specs/pending/` 建立 Spec 文件
2. 使用 `docs/specs/_spec-template.md` 模板
3. 向使用者展示 Spec 內容，等待確認
4. 確認後才開始寫程式碼
5. 完成後將 Spec 移至 `docs/specs/completed/`

## 跨專案改動（額外規則）

若改動涉及多個子專案：
- Spec 必須包含「受影響子專案」表格
- 必須列出建議開發順序
- 每個子專案的受影響檔案必須分開列出

## 豁免

以下不需建 Spec：修正 typo、更新版本號、修改設定值、純格式化調整
```

#### 檔案 3: .claude/commands/_spec-convention.md

```markdown
# 技術規格文件（Spec）撰寫慣例

所有超過 3 行的程式碼改動，在開始寫程式碼之前，必須先建立技術規格文件。

## 強制性規則

1. **超過 3 行的改動 = 必須建立 Spec**
2. **Spec 必須經使用者確認後才能開始寫程式碼**
3. **開發過程中遇到 Bug，必須記錄在 Spec 的 Bug Log 區塊**
4. **開發完成後，Spec 從 `pending/` 移動到 `completed/`**
5. **跨子專案改動，必須列出所有受影響的子專案與開發順序**

### 豁免清單

不需建 Spec：修正 typo、更新版本號、修改設定值、純格式化調整

## 檔案規範

- **位置**: `docs/specs/pending/[task-name].spec.md`
- **命名**: 英文 kebab-case（如 `add-user-crud.spec.md`）
- **模板**: `docs/specs/_spec-template.md`

## 生命週期

1. 建立 Spec → `pending/`（🟡 待確認）
2. 使用者確認（🔵 開發中）
3. 依 Spec 逐步實作
4. 遇問題寫入 Bug Log
5. 完成歸檔 → `completed/`（✅ 已完成）

## Bug Log 格式

| 分類 | 內容 |
| --- | --- |
| **[Bug]** | {問題描述} |
| **[Root Cause]** | {根本原因} |
| **[Solution]** | {解決方案} |
| **[Prevention]** | {預防措施} |

通用 Bug 另存 `docs/bugs/`。
```

#### 檔案 4: .claude/commands/create-spec.md

```markdown
# 建立技術規格文件（Spec）

為即將進行的改動建立技術規格文件。

> **必須遵循** `.claude/commands/_spec-convention.md`

## 執行步驟

### Phase 1: 需求收集
1. 詢問改動內容（功能名稱、目的）
2. 掃描工作目錄，自動辨識受影響的子專案
3. 確認是否有關聯的計劃書（plans/）

### Phase 2: 分析受影響檔案
1. 列出所有會被修改或新增的檔案（按子專案分類）
2. 說明具體的邏輯變更點
3. 評估風險與副作用
4. 依專案依賴關係建議開發順序

### Phase 3: 建立 Spec 文件
1. 使用 `docs/specs/_spec-template.md` 模板
2. 在 `docs/specs/pending/` 建立 `[task-name].spec.md`
3. 填入所有區塊

### Phase 4: 等待確認
1. 展示 Spec 摘要
2. 詢問使用者是否確認
3. 確認後才開始開發
```

#### 檔案 5: .claude/skills/multi-project-workflow.md

```markdown
# Skill: 多專案協作工作流

## 專案架構

| 子專案 | 角色 |
| --- | --- |
| {掃描結果} | {掃描結果} |

## 跨專案關聯性

{依掃描結果推斷，例如：}
- 資料庫結構變更 → 必須同步後端 Model → 可能影響前端 API 呼叫
- 後端 API 變更 → 必須同步前端串接
- 任何資料表新增/修改 → 檢查管理後台是否需要對應的管理介面

## Skill 1: 新增 API Endpoint（全端聯動）

**觸發**: 使用者要求新增 API 或功能需要新的後端接口
**步驟**:
1. 確認是否需要新資料表（→ 先處理資料庫）
2. 在後端子專案新增 Controller / Service / Repository
3. 在前端子專案新增 API 呼叫與對應 UI
4. 確認管理後台是否需要管理頁面
5. 全程遵循 Spec-before-Code

## Skill 2: 新增資料表（資料庫起手）

**觸發**: 使用者要求新增或修改資料表
**步驟**:
1. 在資料庫子專案新增 migration 檔案
2. 在後端子專案新增 Model / DTO
3. 在後端子專案新增 Repository 與 Service
4. 視需求在前端 / 管理後台新增介面
**注意**: migration 順序在 Spec 中明確記錄

## Skill 3: 修復 Bug

**觸發**: 使用者回報 Bug
**步驟**:
1. 分析根本原因（先讀 code，不直接改）
2. 判斷影響範圍（是否跨子專案）
3. 建立 Spec，列出修復方案
4. 使用者確認後修復
5. 更新 Bug Log

## Skill 4: 重構

**觸發**: 使用者要求重構模組或改善品質
**步驟**:
1. 分析問題點
2. 評估影響範圍（grep 所有引用點）
3. 建立 Spec
4. 使用者確認後逐步執行
**注意**: 重構不改變外部行為，Spec 需列出不變性約束

## Skill 5: 更新文件與指引

**觸發**: 新增了 command、功能、或改了架構
**步驟**:
1. 更新 CLAUDE.md
2. 更新 .claude/skills/ 對應技能文件
3. 更新 .claude/commands/ 對應指令（如有）
4. 更新 README.md
5. 格式跟現有慣例一致
```
````

## 日常操作 Prompt 集

以下是導入完成後，日常開發可直接使用的 prompt：

### 跨子專案功能開發

```text
我要新增一個功能：{功能名稱}

請幫我：
1. 先掃描現有子專案結構，判斷哪些子專案會受影響
2. 建立 Spec（docs/specs/pending/{feature-name}.spec.md）
3. 列出每個受影響子專案的檔案與邏輯變更點
4. 建議開發順序
5. 等我確認後再開始開發
```

### Bug 修復

```text
我遇到一個 Bug：{Bug 描述}

請依照以下流程處理：
1. 先分析可能的根本原因（不要直接改 code）
2. 建立 Spec（docs/specs/pending/fix-{bug-name}.spec.md）
3. 在 Spec 中列出：問題描述、重現步驟、根因分析、修復方案、預期測試結果
4. 等我確認後再開始修復
5. 修復完成後更新 Spec 的 Bug Log
6. 若 Bug 有通用參考價值，另存到 docs/bugs/
```

### 分析現有程式碼建立開發慣例

```text
請幫我分析 {資料夾名} 的程式碼，找出：
1. 命名慣例（檔案名、變數名、函式名的 pattern）
2. 目錄結構慣例（檔案怎麼分類放置）
3. 常用的 design pattern（Repository、Service Layer 等）
4. 錯誤處理方式
5. API 回應格式

然後在 .claude/rules/ 建立對應的慣例規則文件，
讓 AI 未來寫程式碼時自動遵循這些慣例。
```

### 新增 command 後更新文件

```text
我新增了一個 command: `xxx`，
請幫我更新 README.md、CLAUDE.md 和 .claude/skills/ 對應文件，
把這個 command 的用途與用法補上去，格式跟現有的一致。
```

### 產生 CLAUDE.md（已有專案但沒有 CLAUDE.md 時）

```text
請幫我掃描這個工作目錄的結構，
依據現有的資料夾、檔案命名慣例、技術棧，
產生一份 CLAUDE.md，包含：

1. 專案概述（一句話描述）
2. 各資料夾用途說明（表格，由掃描結果自動填入）
3. 技術棧與開發環境（從 package.json、*.csproj、requirements.txt 等自動偵測）
4. 開發慣例（命名規則、分支策略、commit 格式）
5. 常用指令（build / test / deploy）
6. 強制性工作流協議（Spec-before-Code）
```
