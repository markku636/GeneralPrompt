# 依據程式脈絡產生指引文件及 Agent Skill

> 使用方式：到目標專案根目錄開啟 Claude Code，複製「一鍵產生」區塊貼上即可。
> 不需要手動填寫任何資訊，AI 會自動掃描專案結構。

---

## 一鍵產生

複製以下內容，貼到 Claude Code：

````text
請幫我掃描這個專案，產生 Claude Code 的指引文件與 Agent Skill。

### 你要做的事

#### Step 1: 掃描專案

分析以下內容，全面理解這個專案：

1. **目錄結構** — 所有資料夾與檔案的組織方式
2. **技術棧** — 從 package.json、*.csproj、*.sln、requirements.txt、go.mod、Dockerfile 等偵測
3. **命名慣例** — 檔案名、變數名、函式名的 pattern（至少抽樣 5-10 個檔案）
4. **Design Pattern** — Repository、Service Layer、MVC、CQRS 等架構模式
5. **API 格式** — 請求/回應的資料結構慣例
6. **錯誤處理** — try-catch 風格、錯誤回傳格式
7. **現有 .claude/ 設定** — 已有的 commands、rules、skills（如果有的話）
8. **Build / Test / Deploy 指令** — 從 scripts、Makefile、CI 設定檔偵測

| --- | --- |
| `{掃描結果}` | {掃描結果} |

## 技術棧
| 項目 | 技術 |
| --- | --- |
| 語言 | {掃描結果} |
| 框架 | {掃描結果} |
| 資料庫 | {掃描結果} |
| 套件管理 | {掃描結果} |

## 開發慣例

### 命名規則
- 檔案名：{掃描結果，如 kebab-case / PascalCase}
- 變數名：{掃描結果，如 camelCase}
- 函式名：{掃描結果}
- CSS class：{掃描結果，如 BEM / Tailwind}

### Design Pattern

## Claude Code 設定

### 自訂指令（.claude/commands/）
| 指令 | 用途 |
| --- | --- |
| `/sync-docs` | 掃描專案，同步更新 README.md 與 CLAUDE.md |

### 常駐規則（.claude/rules/）

| {列出其他已建立的 rules} | |

### 技能模組（.claude/skills/）
{列出已建立的 skills}
```

#### Step 3: 產生 Agent Skill

依據掃描結果，在 `.claude/skills/` 產生技能文件，格式如下：

```markdown
# Skill: {專案名稱} 協作技能

## 專案架構總覽

| 模組/資料夾 | 職責 |
| --- | --- |
| {掃描結果} | {掃描結果} |

## 模組關聯性

{掃描結果，描述模組之間的依賴與資料流向}

## Skill 1: 新增功能

**觸發**: 使用者要求新增功能或頁面
**步驟**:
1. 確認功能涉及哪些模組
2. **⛔ 攔截點 — 先寫 Plan 文件才能寫 Code**（規則見 `.claude/rules/plan-first.md`）
   - 在 `docs/plans/` 建立 `{feature-name}.md`，記錄：目標、影響範圍、方案選擇、檔案變更清單
   - 使用者確認 Plan 後，才可對程式碼檔案執行 Edit / Write
3. 依專案慣例建立對應的檔案（命名、位置遵循現有 pattern）
4. 遵循專案的 design pattern
5. {依掃描結果補充專案特有的步驟}
6. 實作完成後，回頭更新 Plan 文件，標記狀態為「已完成」

## Skill 2: 新增 API Endpoint

**觸發**: 使用者要求新增 API
**步驟**:
1. **⛔ 攔截點 — 先寫 API Spec 文件才能寫 Code**（規則見 `.claude/rules/plan-first.md`）
   - 在 `docs/plans/` 建立 API 規格文件，記錄：endpoint、method、request/response schema、錯誤碼
   - 使用者確認 Spec 後，才可對程式碼檔案執行 Edit / Write
2. {依掃描結果，列出該專案新增 API 的標準步驟}
3. 遵循現有的回應格式與錯誤處理慣例
4. 實作完成後，回頭更新 Spec 文件，標記狀態為「已完成」

## Skill 3: 修復 Bug

**觸發**: 使用者回報 Bug
**步驟**:
1. 分析根本原因（先讀 code，不直接改）
2. 判斷影響範圍
3. 修復後確認不影響其他模組

## Skill 4: 重構

**觸發**: 使用者要求重構
**步驟**:
1. **⛔ 攔截點 — 先寫重構計畫才能改 Code**（規則見 `.claude/rules/plan-first.md`）
   - 在 `docs/plans/` 建立重構計畫，記錄：問題分析、重構策略、影響檔案清單、驗證方式
   - 使用者確認計畫後，才可對程式碼檔案執行 Edit / Write
2. 分析問題點
3. 評估影響範圍（grep 所有引用點）
4. 逐步執行，不改變外部行為
5. 實作完成後，回頭更新計畫文件，標記狀態為「已完成」

## Skill 5: 更新文件

**觸發**: 新增了 command、功能、或改了架構
**步驟**:
1. 更新 CLAUDE.md
2. 更新 .claude/skills/ 對應技能文件
3. 更新 README.md
4. 格式跟現有慣例一致
```

#### Step 4: 產生 Plan-First 攔截規則

在 `.claude/rules/` 建立 `plan-first.md`，物理攔截 Edit/Write，強制先寫文件再寫 Code：

**`.claude/rules/plan-first.md`**：
```markdown
# Rule: Plan First, Code Second — Edit/Write 攔截規則

## 核心原則
在對**程式碼檔案**執行 Edit 或 Write 之前，必須先產生對應的 Plan/Spec 文件並取得使用者確認。
這是硬性規則，不是建議。

## 攔截邏輯

### 當你準備對程式碼檔案執行 Edit 或 Write 時：

1. **檢查** `docs/plans/` 是否已有對應的 plan/spec 文件
2. **如果沒有** → 停止，先建立 plan 文件：
   - 用 Write 工具在 `docs/plans/{feature-name}.md` 寫入計畫
   - 向使用者展示計畫摘要，等待確認
   - 使用者確認後，才可對程式碼執行 Edit / Write
3. **如果已有且已確認** → 正常執行 Edit / Write

### Plan 文件格式

```
# Plan: {功能名稱}

## 狀態：草稿 | 已確認 | 實作中 | 已完成

## 目標
{這次變更要達成什麼}

## 影響範圍
| 檔案 | 變更類型 | 說明 |
| --- | --- | --- |
| {file path} | 新增/修改/刪除 | {說明} |

## 方案
{技術方案描述}

## 驗證方式
{如何確認實作正確}
```

### 判斷程式碼檔案的副檔名
以下副檔名視為程式碼檔案，受此規則攔截：
`.ts`, `.tsx`, `.js`, `.jsx`, `.py`, `.cs`, `.go`, `.java`, `.rb`, `.rs`, `.cpp`, `.c`, `.h`, `.swift`, `.kt`, `.vue`, `.svelte`, `.php`

### 豁免條件（不需要先寫 Plan）
- **Bug hotfix**：使用者明確說「緊急修復」或「hotfix」
- **設定檔**：`.json`, `.yaml`, `.yml`, `.toml`, `.env`, `.config` 等
- **文件檔**：`.md`, `.txt`, `.rst` 等
- **單行修改**：僅修改一行的小幅調整（如修正 typo、調整常數）
- **使用者明確指示跳過**：使用者說「直接改」或「不用寫 plan」
```

同時在 `docs/plans/` 建立一個 `.gitkeep` 確保資料夾存在。

#### Step 5: 產生 Slash 指令

建立以下兩個自動同步指令：

**`.claude/commands/sync-docs.md`**：
```markdown
掃描當前專案的資料夾結構、技術棧、
.claude/commands/、.claude/rules/、.claude/skills/ 的所有檔案，
比對現有 README.md 和 CLAUDE.md，
把缺少或過時的內容補上，保留現有自訂內容不覆蓋，格式跟現有的一致。
```

**`.claude/commands/sync-skill.md`**：
```markdown
掃描當前專案的程式碼結構、commands、rules，
比對 .claude/skills/ 底下現有的技能文件，
把缺少的技能補上、過時的描述更新，
保留現有自訂內容不覆蓋，格式跟現有的一致。
```

---

### 完成後回報

產生完畢後，列出：
1. 偵測到的技術棧摘要
2. 偵測到的開發慣例摘要
3. 產生的所有檔案清單
4. 建議後續可補充的 rules 或 skills
````

## 日常維護 Prompt

導入完成後，日常可使用以下 prompt：

### 新增 command 後同步更新

```text
我新增了一個 command: `xxx`，
請幫我更新 README.md、CLAUDE.md 和 .claude/skills/ 對應文件，
把這個 command 的用途與用法補上去，格式跟現有的一致。
```

### 同步指引文件（也可直接用 `/sync-docs`）

```text
掃描當前專案的資料夾結構、技術棧、
.claude/commands/、.claude/rules/、.claude/skills/ 的所有檔案，
比對現有 README.md 和 CLAUDE.md，
把缺少或過時的內容補上，保留現有自訂內容不覆蓋，格式跟現有的一致。
```

### 同步 Agent Skill（也可直接用 `/sync-skill`）

```text
掃描當前專案的程式碼結構、commands、rules，
比對 .claude/skills/ 底下現有的技能文件，
把缺少的技能補上、過時的描述更新，
保留現有自訂內容不覆蓋，格式跟現有的一致。
```
